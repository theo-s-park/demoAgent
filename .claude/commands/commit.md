커밋 규칙을 준수하면서 git commit을 수행합니다.

## 커밋 메시지 형식

```
{타입}: {제목}

- 추가 설명 1
- 추가 설명 2
```

- 타입과 콜론 사이 공백 없음, 콜론 뒤 공백 1개
- 제목과 본문 사이 빈 줄 1개
- 본문은 `-`로 시작하는 bullet point (선택사항)
- 언어: 한국어 권장, 기술 용어·고유명사는 영어 사용
- 이모지 사용 금지
- 제목은 "무엇을 변경했는가"를 쓴다. 변경의 동기(리뷰 반영, 요청사항 등)는 제목이 아닌 본문에 적는다
  - bad: `refactor: CodeRabbit 리뷰 반영`
  - good: `refactor: ProblemDetail 응답에서 내부 메시지 노출 제거`

## 허용 타입
- `feat:` — 새 기능
- `fix:` — 버그 수정
- `refactor:` — 동작 변경 없는 리팩토링
- `chore:` — 빌드, 설정 등 기타
- `docs:` — 문서 변경
- `test:` — 테스트 추가/수정
- `style:` — 코드 포맷, 세미콜론 등 로직 변경 없는 스타일

## 커밋 분리 원칙

- 타입이 다르면 반드시 분리 (예: `chore` 설정 + `feat` 기능 → 커밋 2개)
- 같은 타입이라도 논리적으로 독립된 작업이면 분리
- 분리 시 의존 순서대로 커밋 (예: 설정 → 기능 → 테스트)

## 제외 대상

- `*.md` 파일은 커밋 대상에서 항상 제외한다

## 절차

1. `git status`와 `git diff HEAD`로 전체 변경 내역 확인 (staged + unstaged 모두 포함)
2. 제외 대상 파일을 걸러낸 뒤, 커밋 대상이 없으면 커밋을 중단하고 "커밋 대상 파일이 없습니다"를 안내한다
3. 커밋 대상이 있으면 커밋 분리 원칙에 따라 변경 내역을 논리적 단위로 분류
4. 각 단위별로 해당 파일만 `git add` → 커밋 메시지 작성 → 커밋
5. 모든 커밋 완료 후 `git log`로 확인
